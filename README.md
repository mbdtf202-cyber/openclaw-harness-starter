# OpenClaw Harness Starter

Starter repository for building reusable OpenClaw harnesses.

This repo is generated by `openclaw harness init` and gives you a small but complete
workspace for writing:

- message scenario harnesses
- channel and plugin contract tests
- CLI stdout/stderr harnesses
- regression suites built on `openclaw/plugin-sdk/test-utils`

It is meant to be edited immediately. The generated files are not placeholders in
the abstract sense; they are real, runnable examples that show the public harness
surface area you can build on.

## Who this repo is for

Use this starter when you are:

- validating routing and outbound delivery behavior
- building a plugin or channel adapter and want contract tests
- testing CLI output, warnings, and failure paths
- packaging a reusable harness repo for your own team

If you just need a single test inside an existing OpenClaw plugin repo, you can
copy patterns from here instead of using the whole starter.

### File-by-file guide

#### `README.md`

The main operator guide for this starter:

- what the repo is for
- what each file does
- how to run the examples
- how to replace the demo cases with your own harnesses

#### `package.json`

Defines the project as a lightweight TypeScript + Vitest repo.

Important details:

- depends on `openclaw` so you can import the public harness APIs
- keeps the script surface intentionally small
- defaults to `pnpm test`

#### `tsconfig.json`

Gives you a strict TypeScript baseline suitable for test-heavy harness projects.

#### `vitest.config.ts`

Minimal Vitest configuration that looks for `src/**/*.test.ts`.

The starter deliberately keeps this file small. If your harness grows, this is
where you would add:

- custom setup files
- coverage settings
- include/exclude rules
- environment overrides

#### `src/message-scenario.test.ts`

The main example of the harness product.

It demonstrates how to:

- create a message scenario harness with `createMessageScenarioHarness`
- define a custom channel id
- resolve a delivery target
- send a message through a fake outbound adapter
- assert both target resolution and final outbound delivery

This is the file to copy first when your real task is:

- testing routing regressions
- testing reply threading semantics
- testing target normalization rules
- testing adapter behavior without a live provider

#### `src/cli-output.test.ts`

Shows the CLI harness side.

It demonstrates how to:

- capture stdout and stderr in one helper
- test command output without rewriting `process.stdout.write` spies in every file

Use this as the starting point for:

- human-readable CLI output tests
- warning and error text assertions
- pipe-close and non-TTY behavior checks

#### `.github/workflows/ci.yml`

A minimal GitHub Actions workflow that runs the harness test suite on push and
pull request.

It exists so the generated repo is useful immediately after pushing to GitHub.

#### `.gitignore`

Covers the normal generated files for a TypeScript/Vitest repo.

#### `CONTRIBUTING.md`

Gives collaborators a minimal contribution contract for this harness repo.

#### `CHANGELOG.md`

Provides a simple release history so the repo can evolve beyond a one-off demo.

#### `LICENSE`

MIT license so you can publish or fork the harness repo without extra setup.

## Public OpenClaw APIs used here

This starter is built on the public harness surface exported by:

- `openclaw/plugin-sdk/test-utils`

The most relevant APIs are:

- `createMessageScenarioHarness`
- `createTestRegistry`
- `createOutboundTestPlugin`
- `captureProcessWrites`
- `runRegisteredCli`
- `runRegisteredCliCaptureOutput`

The goal is to keep your harness repo on top of public APIs, not OpenClaw internals.

## Quick start

```bash
pnpm install
pnpm test
```

If you want to create a fresh repo from OpenClaw directly:

```bash
openclaw harness init my-harness
cd my-harness
pnpm install
pnpm test
```

## What the two example tests actually prove

### Message scenario example

The demo verifies that a fake outbound adapter can:

- normalize a target
- send text through a deterministic path
- expose the resolved target in `targetResolutions`
- expose the actual delivery in `deliveries`

That pattern is useful whenever you care about behavioral semantics more than
real network I/O.

### CLI output example

The demo verifies that you can capture:

- stdout
- stderr
- output ordering in a simple command-like flow

This is useful for CLI commands where the rendered output is part of the contract.

## How to customize this repo

The normal progression is:

1. Keep `package.json`, `tsconfig.json`, and `vitest.config.ts`
2. Replace the demo tests in `src/`
3. Add more harness helpers under `src/` as your project grows
4. Expand CI only when the harness needs more realism

Common adaptations:

- Plugin contract repo
  Replace `src/message-scenario.test.ts` with adapter-specific contract suites
- Routing regression repo
  Add multiple scenario tests for threads, targets, and channel policies
- CLI verification repo
  Add command registration and output tests around `captureProcessWrites`

## Suggested growth path

Start small:

- one scenario
- one contract
- one output capture case

Then expand into:

- shared helper modules under `src/`
- per-channel scenario files
- release tagging and coverage gates
- published npm or template distribution

## What this starter deliberately does not include

This repo is intentionally minimal. It does not include:

- a full plugin implementation
- live provider credentials
- Docker or gateway process harnesses
- lint/format stacks beyond what is necessary to run tests

Those can be added later if your harness really needs them.

## CI and publishing

The included GitHub Actions workflow is enough for a template or small internal
tooling repo.

The workflow intentionally avoids lockfile-dependent caching so template users
can push immediately without first committing a generated `pnpm-lock.yaml`.

Typical next steps after generation:

```bash
git init
git add .
git commit -m "Initial harness starter"
gh repo create
git push -u origin main
```

If you want the repo to behave like a reusable team template, enable template
repository mode in GitHub after the first push.

## Dependency policy

This starter pins `openclaw` to `^2026.3.3`.

That gives you a stable public harness API surface while still tracking the same
general release line as the OpenClaw version that generated the repo.

To move forward later:

```bash
pnpm up openclaw
pnpm test
```
